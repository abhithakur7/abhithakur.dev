## Overview

Uplevelit is a multi-tenant SaaS platform serving white-labelled deployments per customer. I lead a team of 5 engineers across backend, frontend, and infrastructure, shipping on a Node.js / TypeScript / NestJS / React stack deployed on GCP.

## What I built

- **Multi-tenant architecture.** Designed RBAC with tenant-level data isolation, so every request resolves to a tenant context before touching data. Wildcard subdomain routing at the edge handles per-tenant white-label branding without per-tenant code branches.
- **Microservices split.** Scoped the platform as a 4-service system behind a dedicated API gateway. Each service owns its domain (billing, tenants, content, notifications) and communicates over typed contracts.
- **End-to-end Stripe billing.** Subscriptions, wallet top-ups, payouts, refunds, and webhook handlers. Idempotent transactions across distributed services eliminate duplicate payments under retry storms and webhook replays.
- **Operational surface.** Containerised services with Docker, deployed via GCR on GCP. CI/CD via GitHub Actions and Bitbucket Pipelines drives production deployments, rollbacks, and monitoring hooks.
- **Integrations.** Twilio for transactional SMS, SendGrid for email, CDN-based media delivery.

## Key decisions

- **Idempotency keys over distributed locks.** Cheaper and safer for Stripe webhooks under retry storms than coordinating locks across services.
- **Runtime tenant isolation over database-per-tenant.** Kept operational overhead flat as tenant count grows; row-level checks enforced in the ORM layer.
- **API gateway as the auth boundary.** Services trust the gateway's signed context — they don't re-verify user sessions. Keeps service-to-service latency low and auth logic single-sourced.

## Role

As team lead, I owned architecture decisions, code review, and delivery alongside hands-on development — particularly the billing and multi-tenant primitives.
